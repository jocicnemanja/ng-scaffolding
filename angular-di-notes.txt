ANGULAR DI: NG_VALUE_ACCESSOR, PROVIDERS, MULTI: TRUE
======================================================

==============================================
1. NG_VALUE_ACCESSOR — INJECTION TOKEN
==============================================

Purpose:
- Bridge between Angular's FormControl/ngModel and a custom form input component.
- Tells Angular: "this component knows how to talk to forms."

The 4 ControlValueAccessor methods:
- writeValue(value)         → form pushes value INTO component
- registerOnChange(fn)      → component pushes value OUT to form
- registerOnTouched(fn)     → component notifies form on blur
- setDisabledState(bool)    → form disables/enables component

Example registration:
  providers: [{
    provide: NG_VALUE_ACCESSOR,
    useExisting: forwardRef(() => MyInputComponent),
    multi: true
  }]

Key rules:
- multi: true is MANDATORY (NG_VALUE_ACCESSOR is a multi-provider).
- forwardRef is needed because the class isn't defined yet when providers array runs.
- useExisting (not useClass) so the form talks to the SAME instance Angular created.

Wiring flow:
  <app-my-input [formControl]="ctrl"></app-my-input>
  1. formControl directive looks up NG_VALUE_ACCESSOR on host.
  2. Finds your component (registered via multi: true).
  3. Calls writeValue() when form pushes new value.
  4. Component calls onChange callback when user types → pushes value back.

Use it for:
- Custom form inputs (toggles, dropdowns, color pickers)
- Wrapping third-party widgets (date pickers, rich-text editors)
Skip it if you don't need form integration.


==============================================
2. EXPLAINING THE PROVIDER (LIKE I'M FIVE)
==============================================

  providers: [{
    provide: NG_VALUE_ACCESSOR,
    useExisting: forwardRef(() => MyInputComponent),
    multi: true
  }]

Analogy: Angular is a mailroom; forms are letters needing delivery.

provide: NG_VALUE_ACCESSOR
  → "I'm registering for a job called 'Form Value Translator'."

useExisting: forwardRef(() => MyInputComponent)
  → "The worker IS me, this same component instance."
  - useExisting = reuse the existing instance, don't create a new one.
    (useClass would create a SECOND copy and form would talk to wrong one.)
  - forwardRef solves chicken-and-egg: the class isn't defined yet
    at the line where you reference it. forwardRef says "look it up later."

multi: true
  → "Add me to the LIST of translators, don't replace the whole list."
  - multi: false → "I am THE translator" (replaces everyone)
  - multi: true  → "I am A translator, add to team"

One sentence:
  "Angular, register this same component instance (use existing, look up later)
   as one of the (multi: true) form-value translators (NG_VALUE_ACCESSOR)."


==============================================
3. PROVIDERS — THE CORE CONCEPT
==============================================

A provider = a recipe telling Angular's DI: "when someone asks for X, here's how."

Long form:
  { provide: TOKEN, useXxx: VALUE }
  - provide = the KEY someone asks for (the "what")
  - useXxx  = HOW to produce it       (the "how")

The 4 flavors of "how":

  1. useClass    — create an instance of this class
     { provide: Logger, useClass: ConsoleLogger }

  2. useExisting — reuse something that already exists (alias)
     { provide: Logger, useExisting: ConsoleLogger }

  3. useValue    — hand over this literal value
     { provide: API_URL, useValue: 'https://api.example.com' }

  4. useFactory  — call this function to produce it
     { provide: Logger, useFactory: (env) => env.prod ? new ProdLogger() : new DevLogger(),
       deps: [ENV] }

Shorthand:
  providers: [UserService]
  ≡ providers: [{ provide: UserService, useClass: UserService }]

WHERE you put providers = SCOPE:
  - @Injectable({ providedIn: 'root' })  → app-wide singleton (default for services)
  - Component providers                  → one instance per component instance
  - Route providers                      → shared by route + children
  - bootstrapApplication() providers     → app-wide global config

Tokens (the "key"):
  - A class:          provide: UserService
  - An InjectionToken: for non-class things (strings, configs, functions)
      export const API_URL = new InjectionToken<string>('API_URL');
      providers: [{ provide: API_URL, useValue: 'https://...' }]
      // inject:
      private apiUrl = inject(API_URL);

Lookup walks UP the injector tree:
  current component → parent → ... → root → "No provider for X" error.


==============================================
4. MULTI: TRUE — REAL-WORLD USES
==============================================

multi: true = the token holds an ARRAY everyone can APPEND to.

----- USE CASE 1: HTTP Interceptors -----
  providers: [
    { provide: HTTP_INTERCEPTORS, useClass: AuthInterceptor,    multi: true },
    { provide: HTTP_INTERCEPTORS, useClass: LoggingInterceptor, multi: true },
  ]
Angular runs ALL interceptors in order on every request.

----- USE CASE 2: NG_VALUE_ACCESSOR -----
Custom form inputs (covered above).

----- USE CASE 3: NG_VALIDATORS -----
A component can be its own validator:
  providers: [
    { provide: NG_VALUE_ACCESSOR, useExisting: forwardRef(() => Cmp), multi: true },
    { provide: NG_VALIDATORS,     useExisting: forwardRef(() => Cmp), multi: true },
  ]

----- USE CASE 4: APP_INITIALIZER -----
Run multiple init steps before app boots:
  { provide: APP_INITIALIZER, useFactory: loadConfig,       deps: [HttpClient],  multi: true },
  { provide: APP_INITIALIZER, useFactory: loadFeatureFlags, deps: [FlagService], multi: true },
Angular waits for ALL to resolve before bootstrap.

----- USE CASE 5: Custom Plugin / Extension Point -----
Dashboard widgets that features register independently:

  export const DASHBOARD_WIDGET = new InjectionToken<DashboardWidget>('DASHBOARD_WIDGET');

  // Each feature registers its widget:
  { provide: DASHBOARD_WIDGET, useValue: { title: 'Users',  component: UsersWidget },  multi: true }
  { provide: DASHBOARD_WIDGET, useValue: { title: 'Orders', component: OrdersWidget }, multi: true }

  // Dashboard consumes the list (doesn't know who registered):
  widgets = inject(DASHBOARD_WIDGET);   // DashboardWidget[]

  Template:
  @for (w of widgets; track w.title) {
    <section>
      <h2>{{ w.title }}</h2>
      <ng-container *ngComponentOutlet="w.component" />
    </section>
  }

----- USE CASE 6: Route-scoped contributions -----
  {
    path: 'admin',
    providers: [
      { provide: DASHBOARD_WIDGET, useValue: {...}, multi: true },
    ],
    loadChildren: () => import('./admin/routes')
  }
Widget exists only while in /admin.

How to consume:
  // modern
  private widgets = inject(DASHBOARD_WIDGET);   // T[]
  // classic
  constructor(@Inject(DASHBOARD_WIDGET) private widgets: T[]) {}

GOTCHA: You can't mix multi and non-multi providers for the same token.
        Forgetting multi: true on one will throw at runtime.

When to reach for multi: true yourself:
- Designing an EXTENSION POINT (features plug in without consumer knowing them)
- COLLECTED BEHAVIOR (interceptors, validators, init hooks, plugins)
- ROUTE-SCOPED contributions to a shared list


==============================================
TL;DR
==============================================

- A provider = recipe: { provide: KEY, useXxx: HOW }
- Listing [Service] is shorthand for useClass: Service
- Where you put it controls SCOPE (app vs component vs route)
- multi: true → token becomes a LIST everyone can append to
- InjectionToken is the key when the thing isn't a class
- NG_VALUE_ACCESSOR is a multi-provider token used to register
  custom form-input components with Angular's forms system
- forwardRef defers class lookup to break the chicken-and-egg with providers
- useExisting reuses the instance; useClass creates a new one
